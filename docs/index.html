How the privacy works

Privacy-first is not a marketing claim. Here's exactly what the code does — and doesn't — do.

✓ Session IDs

A random UUID is created in sessionStorage and used only to deduplicate rapid repeat pageviews within a single tab session. It dies when the tab closes, is never sent to a third party, and is never stored in a database column that could be queried to reconstruct a user's journey across sessions.

✓ Bot detection

The JavaScript layer measures browser capabilities (does navigator.webdriver exist? did the page load in under 50ms?) and sends a compact bitmask — not raw values, not a fingerprint. The PHP layer checks only two HTTP request headers (Accept-Language, Accept) and the User-Agent string. REMOTE_ADDR is never read or passed.

✓ URL sanitization

Before a page URL is stored, it is scrubbed server-side. Any query parameter longer than 40 characters or resembling an email address is stripped. This prevents accidental storage of password-reset tokens, user IDs, or email addresses that sometimes appear in URLs.

✓ Referrer handling

Referrer URLs are reduced to domain-only before storage. The path and query string — which can contain user-identifying tokens from the referring site — are discarded immediately and never written to the database.

✓ Audience data

Operating system and browser are parsed from the User-Agent string. Screen resolution is reported directly by the browser. These are aggregated counters — no row in any table links a device spec back to a specific visit or session.

✓ Heatmap coordinates

Click and scroll coordinates are stored as (page, x_pct, y_pct, element) tuples where x and y are percentage positions on the page. There is no timestamp, session ID, or any field that could link a click to a particular visitor.

Want to verify it yourself? The plugin is fully open source. Run grep -rn "REMOTE_ADDR\|setcookie\|_COOKIE" includes/ assets/ on the source and you'll get zero matches. View source on GitHub →
