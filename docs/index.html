How the privacy works

Privacy-first is not a marketing claim. Here's exactly what the code does — and doesn't — do.

✓ Do Not Track & Global Privacy Control

If a visitor's browser sends a DNT: 1 header or has the Global Privacy Control signal set (navigator.globalPrivacyControl), the tracker script exits immediately — before collecting any signal, before creating a session ID, before sending a single byte. No data is recorded for that visit.

✓ Session IDs

A random UUID is created in sessionStorage and used only to deduplicate rapid repeat pageviews within a single tab session. It dies when the tab closes, is never sent to a third party, and is never stored in a database column that could be queried to reconstruct a visitor's journey across sessions.

✓ No IP addresses, ever

IP addresses are never read, stored, or passed to any function — including the bot detection pipeline. The bot scorer works from a browser-side bitmask (pass/fail scores only, never raw values) plus two request headers; REMOTE_ADDR is not accessed anywhere. Run grep -rn "REMOTE_ADDR" includes/ assets/ on the source — zero matches.

✓ No cookies or persistent identifiers

The plugin sets no cookies and writes nothing to localStorage. There is no browser fingerprinting and no device ID generation. setcookie() and $_COOKIE are not used anywhere. Visitor state within a tab uses sessionStorage only — the browser clears it when the tab closes. The only value that carries across page loads is an optional UTM attribution, also in sessionStorage.

✓ URL & referrer sanitization

Before a page path is stored, it is scrubbed server-side: any query parameter longer than 40 characters or resembling an email address is stripped — preventing accidental storage of password-reset tokens or email addresses that sometimes appear in URLs. Referrer URLs are reduced to domain-only before storage; the path and query string (which can contain user-identifying tokens from the referring site) are discarded immediately and never written to the database.

✓ Audience data

Operating system and browser are parsed from the User-Agent string. Screen resolution is reported directly by the browser. These are aggregated counters — no row in any table links a device spec back to a specific visit or session.

✓ Heatmap coordinates

Click coordinates are stored as (page, x_pct, y_pct, element) records where x and y are percentage positions. Individual records include a tab-scoped session UUID and a timestamp, but session UUIDs are stored in sessionStorage — they expire when the tab closes and cannot identify a visitor across sessions or devices. The heatmap overlay aggregates records by coordinate bucket, discarding session linkage entirely for display.

✓ No cross-site tracking

The tracker runs entirely within your WordPress site. No script is loaded from an external domain, no beacon is sent to a third party, and no data leaves your server. This also means no ad-network leakage and no exposure from a CDN compromise.

Want to verify it yourself? The plugin is fully open source. Run grep -rn "REMOTE_ADDR\|setcookie\|_COOKIE" includes/ assets/ on the source and you'll get zero matches. View source on GitHub →

Accessibility-first design

The plugin introduces nothing that degrades the experience for visitors or site administrators. No popups, no overlays, no forced consent dialogs.

✓ No consent popup required

Because Rich Statistics collects no PII and sets no cookies, most sites do not need a consent banner for analytics at all. Do Not Track and Global Privacy Control are honoured automatically — no configuration required.

✓ Keyboard-navigable, screen-reader compatible

The dashboard uses WordPress core UI components (WP_List_Table, form-table, standard notices) following WordPress accessibility guidelines. Every control is keyboard-navigable and screen reader compatible. Each analytics page has a built-in Help tab explaining the data — no need to leave the admin.

✓ Zero front-end DOM changes

The tracker JavaScript is completely silent and invisible. It adds no elements to your page — no banners, no overlays, no notification bars. Your theme's layout and your visitors' browsing experience are completely unaffected.

Multilingual & international-ready

Built for sites that serve a global audience — with first-class translation support, per-visitor language reporting, and timezone-aware dashboards.

Fully translatable (i18n)

Every string in the admin uses WordPress i18n functions and the plugin ships a rich-statistics.pot template file. Translations can be contributed via GlotPress on WordPress.org or loaded from your own .mo files placed in wp-content/languages/plugins/.

Visitor language breakdown

The tracker captures navigator.language — the browser's declared language preference — on every pageview. The Audience view groups your visitors by language code so you can see which locales your content serves most.

RTL layout support

The admin dashboard stylesheets respect WordPress's RTL mode. The layout flips correctly for right-to-left locales (Arabic, Hebrew, Persian, etc.) without any extra configuration.

Timezone-aware reporting

The tracker records each visitor's declared timezone. Reporting in the WordPress admin uses your site's configured timezone so daily charts always show your local time, not UTC.

Rich Statistics App

Access your analytics from any device — phone, tablet, or desktop. Install it as a Progressive Web App for a native-like experience with offline support.

Open Web App ↓ Linux Desktop (x86_64) ↓ Linux Desktop (ARM64)

Linux Desktop App Native · No Electron

A native WebKitGTK desktop window — built with Tauri. No bundled browser, no Electron overhead. Required libraries install automatically from your package manager.

Debian · Ubuntu · Raspberry Pi OS 
sudo apt install ./rich-statistics-linux-amd64.deb
# ARM boards (Pi 4 / Pi 5):
sudo apt install ./rich-statistics-linux-arm64.deb
Fedora · RHEL 
sudo dnf install webkit2gtk4.1
# extract & run the binary from the .deb
Arch Linux 
sudo pacman -S webkit2gtk-4.1
# extract & run the binary from the .deb
1

Install the plugin

The PWA and desktop app are free. To connect them to your site, you need the Rich Statistics plugin with the Premium plan active — this enables the REST API they use to fetch your analytics.

2

Generate an App Code

In WordPress, go to Users → Profile → Rich Statistics App and click Generate App Code. You'll get a 6-digit code valid for 15 minutes.

3

Open the app & add your site

Open the web app (or the PWA you installed), tap Add Your Site, enter your site URL and the 6-digit App Code, then create an Application Password to complete the connection.

The app connects to your site's REST API using WordPress Application Passwords — no extra accounts, no new services, no data leaves your server.
